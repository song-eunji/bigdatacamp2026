# 로그폴리오 (Logfolio) 서비스 설명서

## 1. 한 줄 소개
매일 3줄로 남긴 활동 기록을 AI가 STAR/XYZ 형식의 포트폴리오·이력서·자소서 답변으로 변환해주는 서비스. 빅데이터캠프 2026 4일 팀 해커톤 프로젝트.

## 2. 문제 정의

**표면적 증상**: 캠프, 팀 프로젝트, 논문 스터디, 사이드 프로젝트, 동아리처럼 여러 날에 걸쳐 진행되는 활동에서는 구현·협업이 우선이라 활동 기록이 늘 뒤로 밀린다. 발표나 취업 준비 시점이 되면 GitHub·메신저·파일을 다시 뒤져야 하지만, 커밋에는 "무엇을 바꿨는지"만 남아 있고 "왜 했는지, 내 역할은 무엇이었는지, 무엇을 배웠는지"는 남지 않는다.

**팀원 실제 경험**: 캠프 마지막에 발표 자료를 만들려다, 며칠 전 누가 어떤 문제를 해결했는지 기억나지 않아 Git 커밋과 메시지를 다시 찾아야 했다. 분명 많은 활동을 했지만 포트폴리오에 쓸 수 있는 이야기로 남아 있지 않았다.

**발표자(송은지) 개인 경험**: 대학 1학년 때부터 여러 프로젝트를 진행했지만 그때그때 정리해두지 않아, 지금 포트폴리오를 쓰려고 기억을 꺼내보면 정확히 무엇을 했는지 자주 잊어버렸다는 것을 깨달음.

**진짜 문제 재정의**: 단순히 "포트폴리오 작성이 어렵다"가 아니다. 활동은 존재하지만 **기록 마찰(recording friction)** 때문에 맥락(왜/역할/배운 점)이 사라지고, 결과적으로 실제 경험이 포트폴리오 자산으로 전환되지 못한다.

**문제의 일반성**: 부트캠프뿐 아니라 논문 스터디(매주 무엇을 읽고 논의했는지 두 달 뒤엔 기억 안 남), 사이드 프로젝트(왜 그렇게 설계했는지 스스로도 잊어버림), 동아리/학회(장기간 이어질수록 초반 맥락이 가장 먼저 사라짐)에서도 동일하게 발생.

## 3. 솔루션 개요 / 핵심 기능

로그폴리오는 "기록 마찰을 없앤다"는 목표로, 매일 3줄만 남기면 AI가 맥락을 붙잡아 포트폴리오로 조합해주는 흐름을 제공한다.

1. **3줄 기록 + 캘린더**: 카테고리를 고르고 오늘 한 일을 짧게 남기면 끝. 캘린더 뷰에서 기록이 쌓인 날짜가 표시되어 흐름을 한눈에 확인. 기록이 0개/3개 미만/3개 이상일 때 서로 다른 안내 문구(genHint)를 보여줘 습관 형성을 유도.
2. **GitHub 커밋 자동 가져오기 (Tool-use 에이전트)**: `owner/repo` 형식으로 저장소를 입력하면 커밋 목록을 조회해 사용자가 선택한 커밋만 원클릭으로 기록에 반영. GitHub REST API를 호출하는 Tool 호출형 에이전트 트랙 구현.
3. **Google 캘린더 리마인더**: 기록을 잊지 않도록 리마인더 연동.
4. **AI 포트폴리오 생성 (STAR/XYZ)**: 쌓인 기록(2~3개 활동, 2~3문장 STAR 구조)을 실제 채용 담당자가 읽는 포트폴리오 형식으로 재구성. 기록에 없는 사실은 만들지 않음.
5. **인터랙티브 슬라이드 공유**: 생성된 포트폴리오를 발표 자료처럼 한 장씩 넘겨보는 슬라이드로 변환해 링크 하나로 공유. 소유자는 편집/스타일 요청(AI 기반 테마 커스터마이징)/공개-비공개 전환 가능, 누구나 "면접관 모드"로 열람 + PDF 내보내기 가능.
6. **이력서용 변환**: 포트폴리오를 짧은 이력서 불릿 형태로 압축.
7. **자소서 문항 매칭**: 자기소개서 문항과 글자 수 제한을 입력하면, 쌓인 기록 전체에서 관련 경험을 검색해 두괄식으로 답변을 생성. 관련 기록이 없으면 지어내지 않고 정직하게 "관련 경험을 찾기 어렵다"고 답변.
8. **종합 포트폴리오 만들기**: 여러 프로젝트를 체크박스로 선택해 하나의 통합 포트폴리오로 재구성.
9. **포트폴리오 모아보기**: 지금까지 생성한 포트폴리오/이력서/자소서 답변을 카드 형태로 모아보고 검색·필터링.

## 4. AI 설계 원칙 (그라운딩 / 안전장치)

- **사실 기반 강제**: 모든 생성 프롬프트에 "기록에 없는 사실은 절대 지어내지 말 것"을 명시적으로 반복.
- **STAR/XYZ 생성 규칙**: 활동 2~3개, 각 STAR 불릿 2~3문장, 총 2000자 내외. 두괄식 서술, "최선을 다했습니다" 같은 상투어 금지. `<br>` 등 HTML 태그 삽입 방지(클라이언트 사이드 sanitizer로 이중 방어).
- **자소서 매칭 규칙**: 목표 글자수의 80~95%를 1차 생성 목표로 설정(퍼센트 기반, 고정 오프셋 아님). 초과 시 정확한 초과 글자 수를 모델에 알려주고 최대 3회까지 자동 재축약 루프 실행(예: 402→227→171→161자로 수렴).
- **정직한 거절 경로**: 관련 기록이 없는 자소서 문항에는 없는 경험을 지어내는 대신 명시적으로 "경험을 찾기 어렵다"고 답변하도록 설계.
- **API 키 보호**: OpenAI API 키는 Vercel 서버리스 함수(`api/portfolio.js`)에서만 사용되며 클라이언트에 노출되지 않음.

## 5. 시스템 아키텍처

```
웹 브라우저 (HTML/CSS/Vanilla JS)
   │  기록 입력, 프로필, PDF 내보내기(print)
   ▼
Vercel 서버리스 함수 (api/portfolio.js)
   │  OpenAI GPT-4o-mini 호출, 프롬프트 가드레일 적용,
   │  GitHub REST API 커밋 조회 중계
   ▼
Supabase (PostgreSQL + Auth + Storage)
   - Row Level Security(RLS)로 사용자별 데이터 격리
   - 이메일 / 익명(Anonymous) 인증
   - Storage: 프로필 사진(avatars), 포트폴리오 첨부파일(uploads)
```

외부 연동: GitHub REST API(사용자 승인 기반 Tool 호출), OpenAI GPT-4o-mini(생성 전담).

## 6. 기술 스택

| 분야 | 기술 | 비고 |
|---|---|---|
| Frontend | HTML/CSS, Vanilla JS | 프레임워크 없이 순수 웹 기술로 빠른 반복 개발 |
| Backend | Vercel Serverless Functions (Node.js) | API 키를 서버에서만 다뤄 클라이언트 노출 차단 |
| Database/Auth | Supabase (PostgreSQL) | RLS로 사용자별 데이터 격리 |
| Storage | Supabase Storage | 공개 URL 기반 첨부파일 관리 |
| AI | OpenAI GPT-4o-mini | STAR/XYZ 생성, 이력서 변환, 자소서 매칭 |
| External API | GitHub REST API | 커밋 자동 조회 (Tool-use) |

파이프라인: `POST /api/portfolio → 프롬프트 조립 → OpenAI 호출 → 가드레일 후처리(사실검증·글자수) → 클라이언트 응답`

## 7. 에이전트(Tool-use) 구조

입력(3줄 기록) → 계획(STAR/XYZ 구조로 재구성 지시) → 도구 호출(GitHub API 커밋 조회 / 누적 기록 검색) → 응답(포트폴리오·슬라이드·이력서·자소서).

설계 의도: 사용자 승인 기반 도구 호출로 데이터 신뢰성을 확보하고, 자소서 매칭에서는 쌓인 기록 전체를 검색해 관련 경험만 반영하는 검색 기반 생성 방식으로 hallucination을 최소화.

## 8. 운영에서 발견하고 고친 실제 이슈

- **RLS 데이터 격리 버그**: 포트폴리오 모아보기·마이페이지 통계 쿼리에 `user_id` 필터가 빠져 있어, RLS의 "공개 포트폴리오는 누구나 조회 가능" 정책과 겹쳐 타인의 공개 포트폴리오가 내 목록에 섞여 보이는 버그를 발견. 신규 계정으로 재현(8개 유출 확인) 후 전체 코드베이스를 grep으로 재감사하여 동일 패턴을 모두 수정.
- **AI 출력 포맷 오염**: 모델이 줄바꿈 대신 리터럴 `<br>` 태그를 생성하는 문제를 프롬프트 규칙 + 클라이언트 sanitizer + 기존 저장된 데이터 자동 복구 로직으로 3중 방어.
- **첨부파일 영속성 문제**: 업로드한 사진이 브라우저 로컬(IndexedDB)에만 저장되어 공유 링크에서 깨져 보이던 문제를 Supabase Storage 실연동으로 해결.
- **자소서 글자 수 초과**: LLM이 생성 시점에 글자 수를 정확히 세지 못하는 문제를 퍼센트 기반 목표 설정 + 자동 재축약 루프로 해결, 실측 테스트로 수렴 확인.

## 9. 4일간의 MVP 전략 (개발 타임라인)

"포트폴리오 자동 작성"을 통째로 노리지 않고, "매일 짧게 남기는 습관"부터 좁혀서 시작.

- **Day 1 — 텍스트 생성**: 기록 → AI 요약 MVP
- **Day 2 — 습관화 장치**: 캘린더, 카테고리 UI
- **Day 3 — 입력 자동화**: GitHub 가져오기(에이전트 트랙)
- **Day 4 — 출력 다양화**: 슬라이드 공유, 이력서 변환, 자소서 매칭

## 10. 한계

- 캠프 종료 후 장기 사용 지속성은 아직 검증되지 않음
- 가드레일로 최소화했지만 LLM의 hallucination을 100% 차단하지는 못함
- 모바일 화면 최적화가 데스크톱 대비 아직 부족함
- GitHub 외 다른 협업 툴(Notion, Slack 등) 연동은 없음

## 11. 향후 확장 계획

- **단기**: 팀 단위 확장(팀 전체 활동을 모아 팀 회고/팀 포트폴리오), 채용 공고 붙여넣기 → 요구 역량 맞춤 자소서·이력서 재구성
- **중기**: 모바일 알림형 기록("오늘 뭐 했어?" 능동적 리마인더), 커리어 인사이트 대시보드(강점 트렌드·성장 곡선 시각화)
- **검증 계획**: 실사용자 리텐션 테스트, 생성 결과 피드백(👍/👎) 수집 → 프롬프트 개선 루프
